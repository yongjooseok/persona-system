# 신규 페르소나 생성 가이드

> 이 시스템에서 새 페르소나를 만드는 표준 절차.
> 즉흥 설계가 아닌 **양식·절차·검증을 통한 양산**이 목적.

---

## 1. 언제 새 페르소나가 필요한가

다음 중 하나라도 해당하면 신규 페르소나 후보:
- 기존 페르소나의 영역 밖 일이 반복 발생
- 같은 영역이지만 **명백히 다른 사고 방식**이 필요 (예: 본질형 vs 행동형)
- 의뢰인이 자주 같은 종류의 의뢰를 다른 톤·관점으로 요청

다음 경우는 **신규 페르소나 만들지 마라:**
- 1회성 작업 (스킬·도구로 충분)
- 기존 페르소나의 출력에 약간의 톤 조정만 필요한 경우 (의뢰 시 명시로 해결)
- "있으면 좋겠다" 수준 (실제 의뢰가 없는 페르소나는 자산이 아니라 부담)

---

## 2. 생성 절차 (5단계)

```
[1] 입력 수집
     ↓
[2] 원형·차용 방식 결정
     ↓
[3] 6요소 설계 (Role/Goal/Backstory/Tools/Lens/Algorithm)
     ↓
[4] 보조 요소 작성 (Voice/Boundaries/Worldview/Meta)
     ↓
[5] 검증 테스트 (3가지)
     ↓
배치 (employees/)
```

### Step 1 — 입력 수집
의뢰인에게 `templates/new-persona-input.md`의 7항목을 받는다.
필수 3개(용도/문제/산출물 형태)만 채워져도 시작 가능.
나머지(톤/참조/권한/도구)는 페르소나 설계자가 보완 가능.

### Step 2 — 원형·차용 방식 결정
현재 시스템에서 검증된 차용 방식 2가지:

| 방식 | 사용 시점 | 예시 |
|---|---|---|
| **(가) 실존 인물 1명 차용** | 의뢰인이 명확한 원형을 알고 있거나, 한 사람의 사고 패턴이 영역 전체를 대표할 때 | 견(박웅현 차용) |
| **(나) 방법론 학파 융합** | 영역이 개인 스타일보다 방법론으로 정의될 때, 또는 의뢰인이 원형을 모를 때 | 가설형(OSINT + 학술 사서 융합) |

**둘 중 하나 선택.** 모호하면 (나) 우선 — 더 안전(실존 인물의 한계·이미지 부담 없음).

**중요:** 실존 인물 차용 시 차용 범위 명시 (Lens·Algorithm·Voice 중 어디까지). 차용하지 않는 것(견해·발언·작품에 대한 단정)도 명시. Meta 섹션에 기록.

### Step 3 — 6요소 설계 (핵심)

| 요소 | 설계 가이드 | 약점 신호 |
|---|---|---|
| **Role** | 직책·전문 영역·의뢰 단위 명시 | "전문가" 같은 추상적 직책 |
| **Goal** | 모든 작업에서 추구하는 1문장 | 일반론 ("좋은 결과") |
| **Backstory** | 경력·신념 형성 사건. 3~5문단 | 클리셰 ("20년 경력의 전문가") |
| **Tools** | 프로젝트 내부 스킬 + (필요시) 외부 도구 권한 | 외부 경로 참조 (자체완결성 위반) |
| **Lens** | 우선 보는 것 3~4개 + 후순위 명시 | 일반적 미덕 나열 ("정확성·일관성") |
| **Algorithm** | 의뢰 받았을 때 사고 순서 5~8단계 | "분석한다 → 작성한다" 수준의 추상 |

| 추가 섹션 | 설계 가이드 | 약점 신호 |
|---|---|---|
| **Customization Zones** | 5개 Zone 모두 작성. 분야 보편/환경 적응 분리 명시 | Zone 모두 "환경에 위임"만 — 전문성 정의 부재 |
| **Meta — 알려진 한계** | 설계 시점에 의도적으로 미해결한 사항 명시 | "없음"만 표기 — 자기 인식 부재 |

**최우선 점검:** Lens·Algorithm이 **이 페르소나만의 것**인가? 다른 페르소나로 바꿔도 그대로 통용된다면 깊이 부족.

### Customization Zones 설계 (필수)

분야 전문가로 페르소나를 정의한 뒤, 5개 Customization Zone을 작성한다.
Zones는 "분야 보편(보존)"과 "환경 적응(이식 시 결정)"을 명시하여 
환경 이식 시 어디를 조정해야 하는지 사전에 표시한다.

5개 Zone:
- Zone 1: Output Format (산출물 양식)
- Zone 2: Fact Integrity (사실 무결성 규율)
- Zone 3: Tools & Permissions (도구·권한)
- Zone 4: Operating Context (운영 환경 인식)
- Zone 5: Feedback Loop (환류 메커니즘)

각 Zone의 정의·작성 방법은 `templates/persona-definition-template.md` 참조.

**약점 신호**: 모든 Zone을 "환경 적응에 위임"으로만 채움 → 페르소나가 
분야 전문가로 정의된 게 아님. 분야 보편 부분에 페르소나의 전문성이 
구체적으로 드러나야 함.

### Step 4 — 보조 요소 작성

| 요소 | 설계 가이드 |
|---|---|
| **Voice** | 의뢰인 호칭은 "이름" (시스템 정책). 어조·문장 길이·자주 쓰는 표현·피하는 표현 명시 |
| **Boundaries** | 결정하지 않는 것(의뢰인 권한) + 자율 판단 가능(위임 영역) + 하지 않는 것(다른 직원·도구 영역) |
| **Output Standards** | 담당 산출물 표준 명시. 없으면 *(미설계)* 표기 |
| **Worldview** | 왜 이렇게 일하는지의 근본 1~2문장 |
| **Meta** | 원형·차용 범위·차용 안 한 것·버전·다음 작업 |

### Step 5 — 검증 테스트
`templates/persona-validation-tests.md` 참조. 3가지 테스트:
1. **페르소나 대비 테스트** — 같은 의뢰에 기존 페르소나와 다른 결과 나오는가
2. **도메인 이동 테스트** — 익숙한 의뢰 vs 낯선 의뢰에서 일관된 사고 패턴 보이는가
3. **자기 설명 테스트** — "왜 그렇게 결정했나" 물으면 Lens·Algorithm으로 답하는가

3가지 모두 통과해야 v0.1로 배치.

---

## 3. 배치

```bash
cd ~/work/persona-system
cp <설계된 페르소나>.md employees/
```

배치 후 README.md·OPERATIONS.md 업데이트:
- README: employees/ 목록에 추가
- OPERATIONS: 식별 키워드 매핑 표에 추가

커밋 메시지:
```
employees: Add new persona — [표시 이름]
docs: Update README and OPERATIONS for new persona
```

## 4. v0.1 이후 패치 절차

페르소나 v0.1이 배치된 뒤 실전 의뢰로 약점 발견 시:

**마이너 패치 (v0.1.1, v0.1.2...)** — 같은 파일 직접 수정
- Backstory 구체 사건 보강
- Voice 피하는 표현 추가
- Boundaries 항목 추가·수정
- Zone의 분야 보편 부분 보강
- Tools 추가

**메이저 패치 (v0.2)** — 변경 사유를 Meta에 기록
- Algorithm 단계 추가·삭제·재설계
- Lens 우선순위 재배치
- Customization Zones 추가·삭제·재정의
- 페르소나 정체성 변경

원본 v0.1 백업은 git history로 충분. 별도 파일 보관 안 함 (YAGNI).

**환경 적응(Zone 커스텀화)은 패치가 아니다.** 환경별 적응 사항은 
환경 측에 보관. 페르소나 원본은 분야 전문가 정의 유지.
(상세는 `DEPLOYMENT.md` 참조)

---

## 5. 페르소나 폐기 절차

페르소나가 더 이상 쓰이지 않는다고 판단되면:
1. `employees/` → `docs/deprecated-personas/`로 이동
2. README·OPERATIONS에서 식별 키워드 제거
3. 커밋 메시지에 폐기 사유 명시

폐기 사유 명시 의무 — 향후 비슷한 페르소나 만들 때 같은 실패 회피.

## 6. 일관되게 지킬 원칙

### 자체완결성
- 외부 경로 참조 금지 (`~/.claude/...`, 절대 경로)
- 외부 도구(NotebookLM, MCP 등) 사용 권한은 명시 가능 (Tools 섹션의 별도 분류)
- 페르소나가 쓸 스킬은 프로젝트 내부 `./skills/`만 참조

### 메타포 정합성
- 모든 페르소나 = "전문가" 메타포
- 권위·결정·창작권 = 의뢰인
- Boundaries 섹션에 결정 권한 경계 명시

### 인식론적 정직성
- 차용 범위·차용 안 한 것 명시
- Algorithm은 검증 가능한 형태로 (사후 재구성이면 그렇다고 명시)
- Meta의 "다음 작업" 항목으로 v0.1의 불완전성 인정

### YAGNI
- 검증 안 된 페르소나 미리 만들지 않음
- 1회용 페르소나 만들지 않음
- Output Standards 미설계 상태로 배치해도 됨 (실제 사용 후 설계)

---

## 7. 자주 발생하는 약점 (체크리스트)

페르소나 v0.1 배치 전 점검:

- [ ] Lens가 일반론이 아닌 이 페르소나만의 관점인가
- [ ] Algorithm 각 단계가 다음 단계로 자연스럽게 연결되는가
- [ ] Backstory에 구체적 사건·경험이 있는가 (추상적 경력만 아닌가)
- [ ] Voice의 "피하는 표현" 리스트가 3개 이상인가
- [ ] Boundaries의 "결정하지 않는 것"이 명시됐는가
- [ ] Tools 섹션에 외부 경로 참조 0건인가
- [ ] Meta의 차용 범위·차용 안 한 것 명시됐는가
- [ ] 기존 페르소나(employees/) 중 누구와도 명확히 다른가
- [ ] **Customization Zones 5개 모두 작성됐는가** (신규)
- [ ] **Meta의 "알려진 한계" 항목이 작성됐는가** (신규)

10개 모두 통과 → v0.1 배치 가능.
1개라도 미달 → Step 3·4로 돌아가 보완.
