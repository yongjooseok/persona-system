# 페르소나 운영 방식 비교 — Q1 결정용 참고서

> 목적: 콘텐츠 전략가를 어떤 레이어로 구현할지 결정하기 위한 비교 자료.

---

## 1. 일반적으로 페르소나는 4가지 방식으로 운영된다

| # | 방식 | 작동 원리 | 사용 예 |
|---|---|---|---|
| ① | **System prompt 주입** | 대화 시작 시 "너는 ~다"로 시스템 프롬프트에 박는다 | ChatGPT Custom GPT, Claude Projects의 Instructions |
| ② | **MD 정의 파일 호출** | 페르소나를 `.md`로 저장, 필요시 `@persona.md`로 컨텍스트에 로드 | Claude Code의 일반적 패턴, Cursor `.cursorrules` |
| ③ | **Sub-agent (에이전트 분리)** | 독립 컨텍스트·도구권한을 가진 별도 에이전트, 메인이 위임 | Claude Code sub-agents, Cline, Roo Code |
| ④ | **Skill로 패키징** | 페르소나 + 절차 + 도구를 하나의 스킬로 묶고 트리거로 활성화 | 이미 보유한 `blog-writer`, `team-builder` 등 |

---

## 2. 각 방식의 트레이드오프

| 기준 | ① System prompt | ② MD 파일 | ③ Sub-agent | ④ Skill |
|---|---|---|---|---|
| **설정 난이도** | 가장 쉬움 | 쉬움 | 중간~어려움 | 중간 |
| **재사용성** | 낮음 | 높음 | 높음 | 매우 높음 |
| **컨텍스트 격리** | 없음 | 없음 (오염됨) | **있음** | 부분적 |
| **직원 간 협업** | 어려움 | 사용자가 직접 전환 | 메인이 자동 위임 | 트리거 의존 |
| **버전 관리** | 어려움 | Git 가능 | Git 가능 | Git 가능 |
| **병렬 작업** | 불가 | 불가 | **가능** | 부분 가능 |

핵심 차이는 **"컨텍스트 격리"와 "자동 위임"**. 직원이 1명일 땐 의미 없고, 팀이 되는 순간 결정적 차이가 된다.

---

## 3. 콘텐츠 전략가에 4가지를 적용하면

**① System prompt 방식**
```
[대화 시작 시 매번]
"너는 박웅현 스타일의 콘텐츠 전략가다. REF/RFE를 활용해..."
```
→ 매번 복붙. 다음 직원 부르려면 새 대화 시작. **탈락 후보.**

**② MD 정의 파일 방식**
```
~/.claude/employees/content-strategist.md  ← 직원 정의
~/.claude/personas/박웅현-style.md         ← 페르소나
```
대화에서: `"@content-strategist 이 주제 다뤄줘"`
→ Claude가 정의 파일을 읽고 그 직원으로 응답. 같은 대화에서 다른 직원으로 전환하려면 사용자가 명시.

**③ Sub-agent 방식**
```
~/.claude/agents/content-strategist/
  ├── system.md      ← 페르소나
  ├── tools.json     ← 허용된 스킬·도구
  └── workflow.md    ← 표준 절차
```
대화에서: 메인 Claude가 자동으로 "이건 콘텐츠 전략가 일이다" 판단하고 위임. 결과만 받음.
→ 강력하지만, **Claude Code의 sub-agent 기능 정확한 사양 확인 필요** (추정 영역).

**④ Skill 방식**
```
~/.claude/skills/content-strategist/SKILL.md
```
키워드 트리거로 활성화. 이미 운영 중인 `blog-writer`와 같은 형태.
→ 페르소나의 "관점"보다는 "절차"가 강조되는 한계. 직원 개념엔 살짝 빈약.

## 4. 추천 — 단계적 진화 경로

```
[1차 시범] ②MD 정의 파일
  ↓ (콘텐츠 전략가 1명으로 검증)
[2차 확장] ②MD + ④Skill 보완
  ↓ (직원 2~3명, 산출물 표준 안정화)
[3차 본격] ③Sub-agent로 마이그레이션
  ↓ (팀 오케스트레이션, 병렬 작업)
```

**1차에 ②를 권하는 이유:**
- 설정이 가장 가볍다 → 페르소나·스킬·산출물 표준 조합 자체를 먼저 검증
- 잘못 설계해도 `.md` 한두 줄 고치면 끝
- 기존 스킬(`team-builder`, `content-os` 등)을 그대로 호출 가능
- 작동이 검증된 뒤 sub-agent로 옮겨도 정의 파일이 그대로 자산이 됨

**③을 1차에 안 쓰는 이유:**
- sub-agent 설정·디버깅 비용이 직원 1명 검증에는 과함
- Claude Code의 sub-agent 사양에 의존하므로 변경 리스크 있음 (추정)

---

## 5. 불확실 지점 (명시)

- Claude Code sub-agents의 **2026년 5월 현재 정확한 사양**은 확인 필요. 자동 위임 동작 방식, 컨텍스트 전달 범위, 도구 권한 격리 수준 — 일반론으론 답할 수 있으나 로그의 Claude Code 버전·설정에서 정확히 어떻게 도는지는 직접 확인 권장.
- "일반 사용자들이 어떻게 운영하는가"는 일부 추정 포함. 다만 4가지 방식 자체는 LLM 도구 생태계에서 표준화된 패턴.

## 6. 다음 결정

위 표를 본 뒤 로그가 정해야 할 것:

- **A안 채택:** ② MD 정의 파일 + 단계적 진화 → 바로 콘텐츠 전략가 v0 구축으로 진행
- **B안 채택:** ③ Sub-agent로 처음부터 — Claude Code sub-agent 사양을 먼저 검증한 뒤 진행
- **C안:** 다른 조합 / 추가 질문

A안이면 다음 산출물은 **콘텐츠 전략가 직원 정의 파일 초안**과 **첫 산출물 표준** 두 개.
