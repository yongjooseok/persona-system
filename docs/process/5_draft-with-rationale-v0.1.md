# `draft-with-rationale.md` — 산출물 표준 v0.1

> 콘텐츠 전략가 見이 **초안 의뢰**에서 제출하는 표준 산출물.
> 의뢰인이 받자마자 검토·결정·편집할 수 있는 형태로 설계.

---

## 1. 설계 원칙

전문가 메타포에서 산출물은 4가지를 만족해야 한다.

| 원칙 | 의미 |
|---|---|
| **결정 가능성** | 의뢰인이 어디서 결정해야 하는지 명확 |
| **추적 가능성** | 왜 이 초안인지 근거가 따라옴 |
| **수정 가능성** | 의뢰인이 부분 수정·반려·재의뢰 쉬움 |
| **대안 가시성** | 다른 길도 보여줘서 선택 폭 제공 |

**결정·근거·수정·대안.** 네 가지가 빠지면 그냥 "초안 파일"이지 컨설팅 산출물이 아니다.

---

## 2. 견의 Algorithm 8단계 → 산출물 매핑

```
[1] 한 문장 추출      → ## 한 문장
[2] 진부함 나열       → ## 근거 > 안 한 것들
[3] 다른 자리 찾기    → ## 근거 > 끌어온 자리
[4] 연결의 다리       → ## 근거 > 왜 이 표현
[5] 표현 단위 결정    → ## 근거 > 왜 이 길이
[6] 초안 작성         → ## 초안
[7] 자기 검토         → ## 자기 검토
[8] 제출 + 근거       → 산출물 전체
```

산출물이 사고 과정의 거울이 된다. 따라서 견의 사고 순서가 바뀌면 산출물 구조도 바뀐다.

## 4. 채워진 샘플

가상 의뢰 1건으로 채워봤다.

> **의뢰:** Quiet Notes 블로그 'Claude Code 실전 100' 시리즈 첫 글 도입부 (약 300자)

```markdown
# Quiet Notes 'Claude Code 실전 100' 시리즈 첫 글 도입부

## 📌 의뢰 요약
100편 시리즈의 첫 글에 들어갈 도입부. 시리즈 전체의 방향을 한 호흡에 알리는 글.

- **의뢰 형태**: 초안 작성
- **채널·길이**: 블로그(WordPress, Quiet Notes), 약 300자
- **명시 제약**: AdSense 친화적이되 AI 콘텐츠 느낌 안 나게

## 한 문장
> **Claude Code는 코딩 도구가 아니라, 생각을 옮겨놓는 도구다.**

의뢰인의 사용 기록(터미널·Git·문서 작업 분리 없음)에서 추출. 코딩이 본질이 아니라 사고 외부화가 본질이라는 관찰.

## 초안

> 누가 Claude Code를 코딩 도구라고 했는지 모르겠다.
> 처음 만든 사람들도 그렇게 부르긴 했을 것이다.
> 한 달을 쓰며 발견한 것은 그게 아니었다.
> 머릿속에 흩어진 생각을, 손 밖으로 한 줄씩 옮겨놓는 일이었다.
> 지금부터 100번에 걸쳐, 옮겨놓은 흔적을 적는다.

## 근거

### ① 왜 이 표현인가
"코딩 도구"라는 흔한 호칭을 첫 줄에서 부정. 독자가 가져올 클리셰 기대를 끊고 본질로 끌어온다. "옮겨놓는다"는 능동의 약한 동사 — 강조 없이 무게가 실린다.

### ② 왜 이 구조인가
부정 → 의심 → 발견 → 선언. 4박자. 마지막 한 문장이 100편 시리즈의 시작 선언이 된다.

### ③ 왜 이 길이인가
도입부는 한 호흡에 들어가야 한다. 5문장, 약 250자. 더 짧히면 발견의 무게가 줄고, 길어지면 도입이 아니라 본문이 된다.

### ④ 안 한 것들
- "혁신적인 AI 도구"
- "차세대 코딩 환경"
- "AI 시대의 필수 도구"
- "개발자라면 반드시" — 진입 차단
- "여러분도 시작해보세요" — 클리셰 결말

### ⑤ 끌어온 자리
**글쓰기·일기의 영역.** "생각을 옮긴다"는 표현은 본디 글쓰기의 언어. 코딩 영역에 글쓰기 영역의 동사를 가져옴으로써 도구의 정체를 재정의.

## 대안

### 대안 A — 정의 우선형
> Claude Code가 무엇인지 한 줄로 말하면, 코딩 도구가 아니다.
> 머릿속의 생각을 손 밖으로 옮기는 도구다.
> 지금부터 100번에 걸쳐, 그 흔적을 적는다.

### 대안 B — 경험 우선형
> 처음 Claude Code를 설치했을 때, 나는 코드 한 줄을 쓸 줄 몰랐다.
> 두 달 뒤, 나는 코드보다 생각을 더 자주 옮기고 있었다.
> 무엇이 변했는지, 100번에 걸쳐 적는다.

## 자기 검토
- [x] 또렷한가? 예
- [x] 진부함 덜어냈는가? 예 (④ 명단)
- [x] 본질 보이는가? "생각을 옮긴다"로 명시
- [x] 형용사 의심했는가? "흩어진" 1개만 의도적 사용

## 의뢰인 결정 포인트

1. **호칭 통일** — "이것" 같은 지시어 vs "Claude Code" 명시
2. **100번 노출 시점** — 도입부 끝 vs 본문에서 자연스럽게
3. **톤 강도** — 메인안의 담담함 vs 대안 B의 1인칭 경험담

## 다음 의뢰 추천
- [x] **본문 작성** — 도입부 확정 후
- [ ] **시리즈 전체 구조 설계** — 100편의 흐름·키워드 맵
- [ ] **변형** — 같은 본질을 SNS 카드용으로
```

## 5. 변형 가능한 부분 (운영 중 결정)

이 포맷은 v0.1이다. 운영하면서 다음이 바뀔 수 있다.

| 부분 | 가능한 변형 |
|---|---|
| **자기 검토 체크리스트** | 의뢰인 피드백 누적되면 항목 추가·교체 |
| **대안 개수** | 짧은 의뢰엔 1개, 큰 의뢰엔 3개까지 |
| **근거 5항목** | 의뢰 유형에 따라 일부 생략 가능 (예: 검수 의뢰엔 ④⑤만) |
| **결정 포인트 개수** | 작은 의뢰엔 1~2개, 큰 의뢰엔 5~6개 |

핵심 원칙(결정·근거·수정·대안)은 유지하되, **세부 섹션은 의뢰 유형에 맞춰 조절.**

## 6. 한계와 정직한 진단

1. **샘플은 가상 의뢰 기반.** 실제 의뢰로 돌려봐야 포맷 약점이 드러난다
2. **대안 섹션이 무거울 수 있다.** 작은 의뢰엔 과잉. 의뢰 크기 기준의 분기 필요할 수도
3. **자기 검토 체크리스트가 4개뿐.** 견의 Lens·Algorithm 전체를 반영하지 못함. 실전 가동 후 확장 필요
4. **의뢰인 결정 포인트의 질이 가장 중요한데 가장 모호.** 견이 진짜 결정 포인트를 찾는지, 의례적으로 채우는지는 실전에서만 검증됨
