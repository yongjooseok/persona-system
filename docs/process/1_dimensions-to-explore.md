## 차원 ① — 페르소나의 깊이

**핵심 인식:** 페르소나가 얇으면 어떤 운영 방식(MD/Sub-agent)을 써도 결국 같은 답이 나온다.

**깊이의 층위:**
```
표층  →  말투·톤
       ↓
       어휘·비유 패턴
       ↓
       판단 기준 (무엇을 중요하게 보는가)
       ↓
       사고 알고리즘 (어떤 순서로 생각하는가)
심층  →  세계관 (왜 그게 중요한가)
```

**핵심 질문:**
- "박웅현 스타일"이라고 하면 **어느 층위까지** 만드는 건가?
- 톤만 흉내내는 페르소나는 단순 프롬프트로 충분. 사고 알고리즘까지 가야 진짜 다른 사람.
- 페르소나 깊이를 어떻게 검증하나? (같은 입력 → 다른 페르소나가 다른 출력을 내는가)

**로그의 단서:** team-builder 스킬에 이미 "실제 전문가 웹 검색 소싱" 구조가 있음. 이게 표층/심층 어느 정도까지 파주는지 점검 필요.

---

## 차원 ② — 산출물 품질의 정의

**핵심 인식:** "최상의 결과물"의 기준이 정의되지 않으면 직원이 일을 잘했는지 판단할 방법이 없다. 그러면 시스템 개선도 불가능.

**평가 차원 (예시):**

| 차원 | 측정 가능성 | 누가 평가 |
|---|---|---|
| 정확성 (사실 오류 없음) | 자동 가능 | AI 검수자 |
| 일관성 (페르소나 톤 유지) | 일부 자동 | AI 검수자 |
| 독창성 (관점·아이디어) | 어려움 | 사람 |
| 유용성 (목적 달성) | 어려움 | 사람 (사용자) |
| 검수 통과 (편집자 OK) | 가능 | 직원 ↔ 직원 |

**핵심 질문:**
- "최상"의 기준이 **사람(로그)의 직관**인가, **명문화된 체크리스트**인가?
- 직원 산출물이 좋은지 나쁜지 누가 어떻게 결정하나?
- 평가 결과가 다음 작업에 반영되는 루프가 있나?

**로그의 단서:** STATEMENT PDF 포맷이 산출물 표준 1호로 이미 존재. 이게 "품질 기준"인가 "양식 기준"인가? — 둘은 다르다.

## 차원 ③ — 직원의 시간성 (메모리·학습)

**핵심 인식:** 직원이 어제를 기억하느냐가 시스템의 본질을 가른다.

**3가지 모드:**

```
[A] 무기억 직원      — 매번 새 사람, 정의 파일만 동일
[B] 기억하는 직원    — 과거 작업·결정·피드백 누적
[C] 학습하는 직원    — 과거에서 패턴을 뽑아 자기 정의를 갱신
```

**핵심 질문:**
- 콘텐츠 전략가가 지난주에 한 작업을 이번 주에 참조해야 하나?
- 로그가 "전에 그렇게 하지 말랬잖아"라고 했을 때 직원이 알아들어야 하나?
- 기억은 어디에 저장? (컨텍스트 / 외부 DB / RAG / 정의 파일에 누적)

**로그의 단서:** Claude의 memory 시스템이 이미 작동 중 (지금 우리 대화도 그 위에 있음). 직원별 메모리를 분리할 것인가, 공유할 것인가?

## 차원 ④ — 기존 프레임워크의 시사점

**핵심 인식:** 우리가 풀려는 문제는 새롭지 않다. 이미 있는 답을 알면 어느 부분이 우리만의 문제인지 보인다.

**주요 프레임워크 (정리):**

| 프레임워크 | 핵심 구조 | 우리에게 주는 시사점 |
|---|---|---|
| **CrewAI** | Role + Goal + Backstory + Tools | 페르소나 4요소 분해 |
| **AutoGen (MS)** | GroupChat (회의 시뮬레이션) | 수평형 협업 구현 사례 |
| **MetaGPT** | SOP(표준작업절차) 기반 | 산출물 표준 = SOP의 다른 이름 |
| **ChatDev** | 소프트웨어 회사 시뮬레이션 | "AI 회사"의 극단적 사례 |
| **Claude Sub-agents** | 위임형 | Claude Code 네이티브 |

**핵심 질문:**
- 우리가 만들려는 것은 이 중 어디에 가까운가, 아예 다른 무엇인가?
- 이 도구들을 직접 쓰지 않는 이유는? (Claude Code 환경 고정 / 한국어 / 학습 곡선 / 비용?)
- 이들이 풀지 못한 문제를 우리가 풀려는 건가, 같은 문제를 다르게 푸는 건가?

**중요한 추정:** 위 프레임워크들의 정확한 최신 사양은 변동 가능. 비교 진행 시 실제 문서 확인 필요.

## 차원 ⑤ — "직원" 메타포의 적합성

**핵심 인식:** 왜 "직원"인가? 이 메타포가 적합한 작업과 그렇지 않은 작업이 있다.

**메타포별 함의 차이:**

| 메타포 | 함의 | 적합한 작업 |
|---|---|---|
| **도구 (Tool)** | 즉시 호출, 결과만 받음 | 변환·계산·생성 |
| **스킬 (Skill)** | 절차 따라 실행 | 표준화된 작업 |
| **에이전트 (Agent)** | 자율 판단, 도구 선택 | 멀티스텝 작업 |
| **직원 (Employee)** | 역할·책임·관계 보유 | 판단·해석·의견 필요 |
| **동료 (Colleague)** | 대등한 관계, 토론 | 창의·전략 |

**핵심 질문:**
- 로그와 AI의 관계는 **고용주↔직원**인가, **함께 일하는 동료**인가, **위임받는 대행자**인가?
- 콘텐츠 전략가 작업의 70%는 직원 수준이 필요한가, 스킬로 충분한가?
- "직원"이라는 표현이 시스템을 더 좋게 만드나, 메타포에 사로잡혀 과잉 설계하게 되나?

**로그의 단서:** E.OS 프레임워크에 "인간 창작 권위 보호" 원칙이 있음. 이건 "AI = 도구" 관점. 그런데 "직원 시스템"은 "AI = 자율 행위자" 관점. 두 관점이 어떻게 양립하나?


## 어디부터 더 깊이 들어갈까

가장 가치가 큰 순서로 추천:

1. **⑤ 메타포 적합성** — 왜 직원인가? 다른 대안은? *(설계 전체의 방향을 결정)*
2. **① 페르소나 깊이** — 깊이가 없으면 운영 방식 다 무의미 *(가장 실용적)*
3. **② 품질 정의** — 목표 없는 시스템 *(검증 불가능 회피)*
4. **④ 기존 프레임워크** — 바퀴 재발명 방지 *(외부 자산 활용)*
5. **③ 시간성** — 1단계엔 후순위, 2단계에 결정적

---

## 불확실 지점

- 5개 차원 분류는 내가 정리한 것 — 다른 분류도 가능. 빠진 차원이 있을 수 있음
- 차원 ④의 프레임워크별 사양은 최신 검증 필요 (CrewAI·AutoGen 등은 활발히 변화 중)
- 메타포 ⑤의 5분류는 LLM 도구 생태계 일반 분류이지만 학술 표준은 아님
