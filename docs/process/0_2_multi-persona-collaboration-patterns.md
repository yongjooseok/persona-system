# 멀티 페르소나 협업 구조 — 4가지 패턴

> 단일 직원이 아니라 "팀처럼 협업"하는 경우의 설계 옵션.

---

## 1. 협업 패턴은 4가지가 표준이다

| # | 패턴 | 작동 방식 | 비유 |
|---|---|---|---|
| ① | **순차형 (Pipeline)** | A→B→C 직원이 차례로 결과물 전달 | 컨베이어 벨트 |
| ② | **계층형 (Orchestrator)** | 팀장이 작업 분해·분배·통합 | 회사 조직도 |
| ③ | **수평형 (Roundtable)** | 동등한 직원들이 토론·검토·합의 | 회의실 원탁 |
| ④ | **이벤트형 (Event-driven)** | 특정 트리거에 반응 | 알림 받고 출동 |

---

## 2. 각 패턴의 트레이드오프

| 기준 | ① 순차 | ② 계층 | ③ 수평 | ④ 이벤트 |
|---|---|---|---|---|
| **설정 난이도** | 쉬움 | 중간 | 어려움 | 어려움 |
| **자동화** | 부분 | 높음 | 높음 | 매우 높음 |
| **결과 품질** | 일관됨 | 작업따라 강력 | 검수에 강함 | 자동화에 강함 |
| **디버깅** | 쉬움 | 중간 | 어려움 | 어려움 |
| **API 비용** | 낮음 | 중간 | **높음** (다중 호출) | 변동 |
| **로그 케이스 적합도** | ★★★ | ★★ | ★ | ★★ |

---

## 3. 콘텐츠팀에 적용하면

### ① 순차형 (Pipeline) — 가장 단순
```
사용자 요청
  ↓
[리서처] 자료 수집
  ↓
[콘텐츠 전략가] REF/RFE 구조화 + 초안
  ↓
[카피라이터] 톤 정제
  ↓
[편집자] 최종 검수
  ↓
산출물
```
**특징:** 각 직원의 입출력이 명확. 한 단계씩 검증 가능. 사용자가 명시 호출(수동) 또는 자동 핸드오프.

### ② 계층형 (Orchestrator)
```
[콘텐츠 디렉터] ← 요청 받고 작업 분해
   ├─→ [리서처]
   ├─→ [전략가]
   └─→ [카피라이터]
[콘텐츠 디렉터] ← 결과 통합 후 산출
```
**특징:** 디렉터 페르소나가 핵심. 작업 분해·품질 관리 능력 필수. 사용자는 디렉터하고만 대화.

### ③ 수평형 (Roundtable)
```
초안 한 개
  ↓
[전략가] [카피라이터] [편집자] 각자 의견
  ↓
합의 또는 다수결
  ↓
최종본
```
**특징:** 검수·교차 검증에 강함. 비용 크고, 의견 수렴 안 될 수 있음. 비판적 검토가 핵심인 작업(예: 법무, 광고 검수)에 적합.

### ④ 이벤트형
```
[트리거] 새 블로그 글 발행
  ↓
[SEO 직원] 자동 호출 → 메타 태그·키워드 검토
  ↓
[유통 직원] 자동 호출 → SNS 카드 생성
```
**특징:** n8n·zapier 같은 외부 자동화와 결합. 로그가 이미 n8n Daily Routine 운영 중이라 친숙한 패턴.

## 4. 협업 패턴 × 운영 방식 매트릭스

| 협업 패턴 | ② MD 정의 파일 | ③ Sub-agent |
|---|---|---|
| ① 순차형 | **가능** (사용자 수동 호출) | 가능 (자동 핸드오프) |
| ② 계층형 | 어려움 (팀장 자동 위임 한계) | **적합** |
| ③ 수평형 | 매우 어려움 | **적합** |
| ④ 이벤트형 | 외부 자동화 필요 | 외부 자동화 + 직원 호출 |

**핵심 발견:**
- "팀처럼 협업"의 진짜 가치는 ②계층 / ③수평에서 나온다
- 이 둘은 사실상 **Sub-agent 인프라가 있어야 깔끔하게 굴러간다**
- MD 정의 파일만으로는 순차형까지가 한계 (사용자가 수동 호출)

---

## 5. 추천 진화 경로 (수정안)

이전 추천을 멀티 페르소나 협업까지 고려해 갱신:

```
[1단계] ②MD + 단일 직원 검증
  → 콘텐츠 전략가 1명 정의·운영
  → "직원 = 페르소나+스킬+산출물 표준" 공식 검증

[2단계] ②MD + ①순차형 (수동)
  → 콘텐츠팀 3~4명 정의
  → 사용자가 직원 순서대로 호출하며 핸드오프 정의 검증

[3단계] ③Sub-agent + ②계층형
  → 디렉터(팀장) 도입, 자동 위임
  → 이때 Claude Code sub-agent 사양 본격 활용

[4단계] 필요 시 ③수평형 / ④이벤트형 추가
```

## 6. 만약 처음부터 팀으로 가고 싶다면

선택지는 둘 중 하나:

**A. ②MD + 수동 순차형으로 시작**
- 디렉토리:
  ```
  ~/.claude/teams/content-team/
    ├── workflow.md       ← 협업 순서·핸드오프 정의
    ├── researcher.md
    ├── content-strategist.md
    ├── copywriter.md
    └── editor.md
  ```
- 사용자가 `"@content-team 시작"` 또는 직원별 순차 호출
- 장점: 인프라 부담 없음 / 단점: 자동 위임 없음

**B. ③Sub-agent + 계층형으로 시작**
- Claude Code sub-agent 기능 검증 필요 (불확실 지점)
- 디렉터 페르소나가 가장 어려운 설계 — 작업 분해·품질 판단 기준 필요
- 장점: 처음부터 진짜 "팀" / 단점: 디버깅 비용 큼, 첫 직원조차 작동 안 할 수 있음

## 7. 정직한 권고

**"팀처럼 협업"은 콘텐츠 전략가 1명이 작동하는 걸 본 뒤가 안전하다.**

이유:
- 1명도 작동 안 하면 4명이 협업할 수 없음 (당연하지만 자주 무시됨)
- 페르소나·스킬·산출물 표준 공식이 검증되지 않은 상태에서 팀을 설계하면 잘못된 공식을 4번 복제하게 됨
- 협업 패턴(순차/계층/수평)은 1명 검증 후 추가하기 쉬움

**다만 처음부터 팀으로 가야 하는 이유가 있다면** (예: 1명만으로는 산출물 품질이 안 나옴, 팀 자체가 학습 목적):
→ **A안 (②MD + 수동 순차형)** 권장. B안은 sub-agent 사양 검증 후로 미루는 게 안전.

---

## 8. 불확실 지점

- Claude Code sub-agent의 멀티 에이전트 오케스트레이션 정확한 동작은 **검증 필요**
- 외부 프레임워크 옵션(CrewAI, AutoGen, LangGraph 등)은 검토하지 않음. 필요하다면 별도 비교 가능
- 비용 추정은 일반론. 실제 로그의 사용 패턴에서 어떻게 나올지는 실측 필요

- **K3.** 처음부터 팀으로, ③Sub-agent + ②계층형 (B안) → 먼저 sub-agent 사양 검증
