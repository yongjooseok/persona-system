# 전문가 분해 프레임

## 용도
- 프로젝트 시작 시 필요한 관점 도출
- 내가 전문가가 되지 않고 전문가의 질문을 빌림
- Level 3-4 질문을 강제로 끌어냄

---

## 핵심 원칙

```
❌ 내가 공부해서 전문가가 된다
⭕ 전문가의 사고 구조를 질문으로 빌린다
```

---

## STEP 1: 전문가 도출

### 질문
```
"이 프로젝트를 '사람' 기준으로 나누면, 어떤 역할이 필요한가?"
```

### 기준
- ❌ 기술 기준 (프론트엔드, 백엔드...)
- ⭕ 책임 기준 (무엇을 결정하는 사람인가)

### 전문가 수
- 3명 이하: 빠진 관점 있음
- 5명 초과: 범위가 넓음, 분할 필요
- 적정: 3-5명

### 범용 최소 구성 (어떤 프로젝트든)
```
1. 기획자 (무엇을, 왜)
2. 실행자 (어떻게)
3. 검증자 (제대로 됐나)
```

---

## STEP 2: 전문가별 분야 정의

각 전문가가 책임지는 영역을 한 줄로 정의

### 예시: 쇼츠 프로젝트

| 전문가 | 책임 영역 |
|--------|-----------|
| 콘텐츠 기획자 | 무엇을 만들지 결정 |
| 쇼츠 포맷 전문가 | 어떤 구조가 먹히는지 |
| 플랫폼 알고리즘 전문가 | 배포/노출 최적화 |
| 자동화 전문가 | 반복 가능한 시스템 |
| 수익화 전문가 | 돈이 되는 지점 |

### 예시: 블로그 프로젝트

| 전문가 | 책임 영역 |
|--------|-----------|
| SEO 전문가 | 검색 노출 |
| 콘텐츠 전문가 | 글 구조/가독성 |
| 수익화 전문가 | 광고/제휴 배치 |
| 자동화 전문가 | 작성 프로세스 효율화 |

---

## STEP 3: 초고수 질문 5개 (모든 전문가에게 동일)

### 질문 세트

```
1️⃣ 이 분야에서 초보자가 가장 먼저 잘못 붙잡는 것은 무엇인가?

2️⃣ 성과가 나는 사람들은 절대 하지 않는 행동이 무엇인가?

3️⃣ 이걸 시작할 때, 반드시 하나만 고정해야 한다면 무엇인가?

4️⃣ 90%가 중간에 포기하는 지점은 어디이며, 이유는 무엇인가?

5️⃣ 이걸 '이해하지 않아도' 결과가 나오게 하려면 어디까지 자동화할 수 있는가?
```

### 질문 설계 의도

| 질문 | 얻는 것 |
|------|---------|
| Q1 | 피해야 할 함정 |
| Q2 | 하지 말아야 할 것 |
| Q3 | 시작점 고정 |
| Q4 | 포기 방지 전략 |
| Q5 | 자동화 범위 |

---

## STEP 3.5: 모순 해결 (Conflict Resolution)

### 왜 필요한가?
전문가들의 조언이 서로 충돌할 수 있다. 이때 AI가 대신 결정하지 않고, 사용자가 우선순위를 정해야 한다.

### 흔한 충돌 유형

| 충돌 | 전문가 A | 전문가 B |
|------|----------|----------|
| 품질 vs 속도 | "제대로 만들자" | "빨리 내보내자" |
| 기능 vs 단순 | "기능이 많아야" | "심플해야" |
| 투자 vs 절약 | "돈 써서 키우자" | "돈 아끼자" |
| 완벽 vs 출시 | "완벽해질 때까지" | "일단 내보내자" |

### 모순 발견 시 질문

```
"[전문가 A]는 [A 주장]을 말하고,
[전문가 B]는 [B 주장]을 말합니다.
이번 프로젝트에서 딱 하나만 포기한다면, 무엇을 포기하겠습니까?"
```

### 모순 해결 기록

| 충돌 | 선택 | 포기 | 이유 |
|------|------|------|------|
| (기록) | (기록) | (기록) | (기록) |

→ 이 기록은 이후 의사결정 시 참조용으로 사용

## STEP 4: 답변 분류

### 3가지 버킷

각 전문가 답변을 아래로 분류:

```
🟢 내가 직접 해야 할 것
   - 판단, 결정, 창작

🟡 AI/도구에 맡길 것
   - 반복, 정리, 검색, 초안

🔴 아예 안 해도 되는 것
   - 과최적화, 완벽주의, 초기 불필요
```

### 분류 기준

| 조건 | 분류 |
|------|------|
| 내 판단이 필수 | 🟢 직접 |
| 반복/패턴화 가능 | 🟡 AI/도구 |
| "나중에 해도 됨" | 🔴 안 함 |
| "해봐야 모름" | 🟡 테스트 후 결정 |

---

## STEP 5: 실행 여부 결정

### 최종 질문

```
"이 프로젝트는 '배워야만 가능한 구조'인가, '위임하면 바로 굴러가는 구조'인가?"
```

### 판정 기준

| 조건 | 판정 |
|------|------|
| 위임 가능 + 리소스 충분 | ✅ 즉시 시작 |
| 위임 가능 + 리소스 부족 | ⚠️ 축소 후 시작 |
| 학습 필요 + 학습 의지 있음 | ⏸️ 학습 범위 정의 후 시작 |
| 학습 필요 + 학습 의지 없음 | 🛑 중단 권고 |
| 모순 미해결 | 🛑 중단 권고 |
| 6요소 중 3개 이상 모호 | 🛑 중단 권고 |

### 중단 권고 출력

```
🛑 중단 권고

[프로젝트명]은 현재 조건에서 실행을 권장하지 않습니다.

사유:
- [구체적 이유]

선택지:
A. 지금 중단, 조건 바뀌면 재검토
B. 목표를 1/10로 축소하고 시작
C. 제약 조건 변경 후 재분석
D. 그래도 강행 (실패 확률 높음 인지)
```

### 시작 확정 시 출력

```
✅ 시작 확정

[프로젝트명] 실행 조건 충족

🟢 직접: [목록]
🟡 AI/도구: [목록]
🔴 안 함: [목록]

첫 번째 액션: [구체적 행동]
```

## 전체 흐름 요약

```
[프로젝트 주제]
    ↓
[STEP 1] 전문가 3-5명 도출
    ↓
[STEP 2] 각 전문가 책임 영역 정의
    ↓
[STEP 3] 초고수 질문 5개 적용
    ↓
[STEP 3.5] 모순 해결 (충돌 시)
    ↓
[STEP 4] 답변 → 직접/AI/안함 분류
    ↓
[STEP 5] 실행 여부 판정
    ↓
→ ✅ 시작: 실행 단위 확정
→ ⚠️ 축소: 목표 재설정 후 시작
→ 🛑 중단: 조건 변경 시 재검토
```

## 예시: 쇼츠 프로젝트 전체 분석

### 전문가 도출
1. 콘텐츠 기획 초고수
2. 쇼츠 포맷 초고수
3. 플랫폼 알고리즘 초고수
4. 자동화/툴 초고수
5. 수익 구조 설계 초고수

### 초고수 질문 결과 (요약)

| 전문가 | Q1 함정 | Q3 고정점 | Q5 자동화 |
|--------|---------|-----------|-----------|
| 기획 | 트렌드 쫓기 | 타겟 한정 | 주제 리스트 |
| 포맷 | 영상 먼저 | 훅 공식 | 템플릿화 |
| 알고리즘 | 해시태그 집착 | 업로드 주기 | 예약 배포 |
| 자동화 | 완벽한 자동화 | 수동 검수 1회 | 스크립트 생성 |
| 수익 | 조회수 = 수익 | 전환 구조 | 링크 삽입 |

### 분류 결과

```
🟢 직접: 타겟 결정, 훅 검수, 전환 구조 설계
🟡 AI: 주제 리서치, 스크립트 초안, 썸네일 생성
🔴 안함: 해시태그 최적화, 완벽한 편집, 트렌드 추적
```

### 결론
→ 위임 가능 구조
→ 즉시 시작, 스크립트 자동화부터

## 전문가 분해 불가 시 대응

### 증상
- 전문가가 1-2명밖에 안 나옴
- 전문가 영역이 겹침
- "그냥 내가 다 해야 할 것 같음"

### 원인
- 프로젝트 범위가 불명확
- 목적이 혼재

### 해결
1. [6-ELEMENTS.md](6-ELEMENTS.md)로 돌아가서 목적 재정리
2. 범용 최소 구성(기획/실행/검증) 강제 적용
3. 그래도 안 되면 → 프로젝트 분할 필요
